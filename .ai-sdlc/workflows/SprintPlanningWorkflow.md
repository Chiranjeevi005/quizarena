# Sprint Planning Workflow

## Governing Authority
All actions are strictly governed by the AI SDLC Constitution:
[.ai-sdlc/constitution/AI-SDLC-v1.2.md](../constitution/AI-SDLC-v1.2.md)

## Objective
To ensure every engineering sprint is organized around delivering a single complete business capability from end-to-end.

## Sprint Boundaries
Every sprint shall own exactly one primary capability.
A capability may contain multiple features, APIs, pages, services, workflows, validations, and integrations.

## Planning Rules
1. A sprint must not end with partially completed capabilities.
2. No new capability may begin until the current capability satisfies its Definition of Done.
3. Every sprint must have a clear Capability Identity.
4. All planning documentation shall reference capabilities instead of isolated features.

## Definition of Done
A sprint is complete only when:
- Entire capability works end-to-end
- No critical launch blockers remain within the capability
- UI is production-ready
- Security requirements satisfied
- Architecture Review = PASS
- QA Review = PASS
- Documentation updated
- Repository committed

## Capability Checklist
Every capability must maintain a mandatory checklist covering Business, Backend, Frontend, Security, Quality, and Documentation requirements. The checklist must be fully verified before sprint lock.
